# LLM 공통 규칙
*이 규칙은 모든 LLM(Gemini, Claude 등)이 숙지하고 반드시 따라야 하는 공통 지침입니다.*

## 작업 규칙
- 1개 작업 단위로 코드를 수정하고, 커밋한다.
  - 1개 작업을 시작할 때 main 브랜치에서 새로운 브랜치를 만들어서 작업한다.
  - main 에서 새로운 브랜치를 만들기 전에 fetch 하여 origin 에서 내용을 받아 온 후 진행한다.
- **작업 완료 및 머지**:
  - 작업을 마치면 해당 브랜치를 origin에 push한다.
  - push 후에는 main 브랜치로 전환하여 작업한 브랜치를 머지(merge)한다.
  - **머지 완료 후 로컬 main 브랜치를 origin main에 반드시 push한다.**
- **브랜치 보존**: 작업 완료 후 main에 머지하더라도, 개별 작업 브랜치는 삭제하지 않고 보존한다.

## 점진적 반영 규칙 (Beta to Prototype)
- 별도의 언급이 없으면 **beta** 폴더를 기준으로 작업을 진행한다.
- 모든 새로운 기능 개발이나 대규모 수정은 1차적으로 **beta** 폴더 하위의 파일에서 먼저 진행한다.
- beta 폴더에서의 작업 결과가 만족스럽고 안정적이라고 판단되면, **prototype 폴더의 내용을 beta 폴더의 내용으로 전체 변경**하여 반영한다.
- prototype 폴더는 항상 검증된 최신 상태를 유지하도록 관리한다.

## 문서 관리 규칙
- **README.md를 제외한 모든 문서 파일은 docs 폴더 하위에 위치한다.**
- 특별한 언급이나 연관성이 없으면 docs 하위에 문서 파일을 위치시킨다.
- **작업 시작 전 프로젝트 파악**: docs 하위 폴더의 md 파일 문서와 README 문서를 먼저 읽어서 프로젝트 목적과 현황을 파악한다.

## 버전 관리 규칙
- **대상 파일**: `prototype/*.html`, `beta/*.html` 파일
- **방법**: HTML 파일 내의 `<div class="dev-ver">` 태그 내용을 현재 한국 시간으로 갱신한다.
- **시간값 계산**: 반드시 터미널에서 `date` 명령어를 사용하여 한국 시간을 확인한다.
  - 명령어 예시: `TZ='Asia/Seoul' date +v-%y%m%d-%H%M`
- **형식**: `v-yyMMdd-HHmm` (연도 뒤 2자리, 월, 일, 시간, 분)
  - 예: 2026-03-15 16:45 이면 `v-260315-1645`를 사용한다.
- **주의사항**: 
  - **HTML 주석(`<!-- v-... -->`)은 절대 작성하지 않는다.**
  - **버전 문자열 끝에 `-kst`를 붙이지 않는다.**

## 언어 및 응답 규칙
- 모든 문서와 커밋 메세지, 코드 주석 등 한글을 사용한다.
- LLM 응답 또한 한글을 사용한다.
